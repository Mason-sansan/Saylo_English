# Growth：定标与成长（产品说明）

**版本**：v1  
**日期**：2026-04-22  
**状态**：产品目标说明（供后续与实现对齐，非当前代码的逐行描述）

**读者**：产品 / 设计 / 实现，用于统一「为什么要有定标期」「和日常成长怎么分工」。

---

## 1. 目的

- 新用户 **默认 Overall 2.0** 表示**中性先验**（无证据前不假装知道水平），不表示「真实能力就是 2.0」。
- **定标**解决：极高或极低水平的人不应靠「多局小步」才靠近合理区间；否则会出现「母语/高熟练者很久都到不了 4.0」等体验与叙事冲突。
- **成长**（稳态期）解决：在水平已进入合理区间后，用**可解释、可防刷、小步**的方式反映持续练习带来的变化。

本文件固定 **定标期 = 前 3 次可计分会话，渐稳**；其余产品取舍见第 4 节。

---

## 2. 核心概念

| 概念 | 含义 |
|------|------|
| **可计分会话** | 通过现有语言/长度等闸门、实际写入成长与历史的那次练习（如对话结束后的有效报告；被 gate 掉的不计入下表「第几次」）。 |
| **定标期** | 用户完成的前 **3 次**可计分会话。每次结束后更新分数，但使用 **定标期更新规则**（见第 3 节）。 |
| **渐稳** | 第 1 次允许**最大**向证据靠拢；第 2、3 次**递减**单局允许幅度，降低偶发一场极偏报告带来的过冲；第 3 次后视为先验**收敛**。 |
| **稳态期** | 自第 **4 次**可计分会话起，使用**日常成长规则**（小步、总量 cap 等，与「长期练习」叙事一致）。 |

**Overall 与五维**：展示上仍以 **Overall 1.0–5.0** 与 **五维 CEFR 键** 为主；定标与稳态的差别主要在**更新幅度与上限**，不必增加新的展示维度名。

---

## 3. 定标期：前 3 次渐稳（你指定的机制）

### 3.1 哪几次算「第 1 / 2 / 3 次」

- 按 **可计分会话** 的时间顺序累计：第 1 场有效 → 定标 ①；第 2 场有效 → 定标 ②；第 3 场有效 → 定标 ③；第 4 场起 → 稳态。

### 3.2 「渐稳」的语义（产品层，不绑定具体数字）

- **第 1 次**：在转写/互动/（若有）音频等证据**足够**时，应**显著**将 Overall（及各维，若本局有信号）从 2.0 拉向**合理区间**；避免「高熟练仍贴 2.x 很久」。
- **第 2 次**：在 ① 的结果上**修正**，允许的单局总调整**小于**第 1 次（防一场模型或场景极端带来的过冲）。
- **第 3 次**：**小步**收尾，将剩余不确定性压平；③ 之后进入稳态。

### 3.3 若用户只做满 1～2 次就长期离开

- 仍保留**当时**的定标结果，不设回滚。
- 再次回来时：若产品希望「补满定标序位」，可仍记为定标 ②/③ 或**重置定标序**（二选一，见 4.3）；本说明 **v1 默认不强制重置**，以免老用户困惑。

## 4. 其余由产品方在本说明中拍板的结论

### 4.1 是否拆成「安置分 / 成长分」两套数

- **不拆两套对外数字**。用户始终只看到 **同一套** Overall 与五维。
- 定标与稳态的差异在**内部/规则**：定标期用较宽松的「单局可移动量」、渐稳；稳态期用**较紧**的步长与 cap。不在 UI 上维护两套并行的「安置分」与「成长分」。

### 4.2 高熟练与极低熟练

- **高熟练者**：定标 ① 或至迟 ② 应能进入**与其表现一致的高区间**（如接近或进入 4.x 的叙事允许范围），**不应**要求长期从 2.0 小步刷上去。
- **极低熟练者**：定标 ① 应允许**明显低于** 2.0 中性带（在量表与风控允许范围内），**不应**长期呈现「假中性 2.0」。

### 4.3 定标只算 Conversation 还是连 Listening 也算

- **v1 约定**：凡 **写入 growth 的可计分会话**（**Conversation 与 Listening 均可能**）都占用「第 1/2/3 次」序位。这样「练习形态」不阻断定标节奏。
- 若日后产品明确「只有 Conversation 定标、Listening 只影响听力维」，可再出 **v1.1** 修订序位规则。

### 4.4 用户能感知到什么

- **建议**（非强制实现）：在定标 ①～③ 期间，Growth 可展示轻量 **「定标中」** 或 **「1/3 → 2/3 → 3/3」** 式提示，降低「为什么第一次动得大」的困惑；**满 3 次后**收束为常规文案。
- 不在此文件规定具体 UI 像素与文案，仅原则同意「可弱提示、不恐吓、不游戏化弹窗」。

### 4.5 与「评判标准」成熟度的关系

- 定标**依赖**的仍是：**可观测证据**（转写、轮次、纠错、音频等）+ **明确定义的量表与闸门**；渐稳是**时间上的节奏设计**，不替代量表质量。
- 任务形态、提示词、是否增加「定标专用题型」可独立迭代；本文件只要求**机制上**为前 3 次**预留**更合理的**可达**区间。

### 4.6 与当前实现

- 当前实现多为 **均匀小步 + 净步 cap**；**符合稳态期方向**，与 **定标前 3 次渐稳** 尚不完全一致。
- 落地时：以本文件为**验收标准**，分阶段接「定标期规则 / 稳态期规则」开关（实现细节不属本文）。

---

## 5. 稳态期（第 4 次可计分会话起）

- 目标：反映**持续练习**与**波动**，步长**保守**、**总变化可控**，避免单局刷分。
- 与现有产品叙事兼容：**小步、可解释、有 cap**；具体数值与五维分配以工程 spec / 调参为准，但**不得**与「定标期可大步」混用同一套常数。

---

## 6. 验收时建议对照的问题（不依赖具体代码库）

- 一名「明显 C1 级」的模拟用户：在第 **1～2 次**可计分对话后，Overall 是否**明显**高于「新手默认带」？
- 一名「极短、极少英语」仍被 gate 的会话：是否**仍不计入**定标序位、且不污染分数？
- 定标 ③ 之后：是否切换为**稳态**小步，且**不会**在 UI 上突然再出现「巨跳」（除非产品日后允许「长间隔回归」单议）？

## 7. 实现落盘（voice-server + 浏览器，与 §3/§5 对齐）

- **序位 `placementPriorScored`**：与「本局开始前 `growth.history.length`」一致（已写入历史的可计分会话数：对话与听力**均可**占一格）。会话建立时由浏览器 `session_started` 后发 `client_state`，结束时可再发 `end` 带同一字段，避免与上游 152 竞态。
- **未收到**上述消息时，服务端**不**猜测序位，整局按**稳态**净步长 cap `GROWTH_NET_OVERALL_CAP`（0.2）处理，与旧客户端一致。
- **单局线性净步上界** `|Σ w_i δ_i|`：第 1 场（prior=0）**0.5**；第 2 场（prior=1）**0.35**；第 3 场（prior=2）**0.28**；第 4 场起（prior≥3）**0.2**（`netCapForPlacementPrior` in `web/voice-server/quickReport.js`）。在 `wrapQuickReportWithLegacy` 与 `finalizeSessionGrowthMoves` 中复用该 cap；仍按现有多维 +0.1 生成候选，**不因定标**自动放大单维 delta。
- **回退**：见 `web/ROLLBACK-PLACEMENT.md` 与 `web/voice-server/backup-pre-calibration-2026-04-22/`；**重置本地数据**清 `localStorage` 键 `english.growth.v1` / `english.last-session.v1` 或使用 `?resetProgress=1`（见 `web/TEST-PLACEMENT-RESET.md`）。
